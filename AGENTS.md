# AGENTS.md

本文件定义代码代理（Agent）在本仓库中的默认工作规则。目标是帮助代理快速做出正确决策，而不是重复 README。

## 1. 仓库概览

- 本仓库是 `kanDB` 的早期 Rust 工程，目标是构建一个基于 `GPUI` 的桌面数据库查看与管理客户端。
- 当前仓库仍处于 bootstrap 阶段；做实现时要把“早期阶段”作为事实前提，不要把尚未落地的产品能力当成既有基础设施。
- 技术基线：
  - Rust Edition: `2024`
  - 当前 package 名：`kandb`
- 涉及 UI、交互、窗口状态、焦点、异步任务、布局和测试时，优先复用 `.agents/skills/` 中的 GPUI 技能说明，而不是临时拍脑袋决定模式。

## 2. 代码修改原则

- 先做最小变更，再逐步扩展，避免大面积无关重构。
- 保持现有模块边界、命名风格和错误处理方式。
- 新增代码优先复用现有类型、日志模式和公共能力。
- 不要为了“先通过测试”“先修复现象”“先完成功能”而引入 hack、临时绕过或层层兜底来掩盖问题；发现设计、建模、状态流转或架构本身有缺陷时，优先定位并修正根因。
- 不要用额外兜底逻辑把问题隐藏起来；新增兼容或保护分支前，先判断是否是在掩盖上游不正确的数据流、抽象或生命周期设计，并优先修复原有代码。
- 禁止新增或编写 `mod.rs`；Rust 模块入口统一使用与模块同名的 `{module}.rs` 文件。
- 新增依赖必须使用完整版本号，禁止使用宽泛版本写法。
- 涉及行为变更时，至少补充一个对应测试，或明确说明测试缺口。
- 不要修改与当前任务无关的文件格式、导入顺序、目录结构或代码风格。
- 文档与配置文件统一使用 `UTF-8` 和 `LF`。
- 如果修改会影响 README、LICENSE 或仓库公开元数据中的表述，要同步检查并更新，避免代码状态和公开说明脱节。

## 3. UI 与 GPUI 规则

- 默认优先使用简单、可组合的 GPUI 模式；只有在高层 API 明显不够用时，才下沉到更底层的 `Element` 实现。
- 新增 UI 组件时，保持单一职责，避免把布局、状态管理、数据访问和副作用堆在同一个组件里。
- 布局和样式优先遵循 GPUI 的链式风格与主题能力，避免散落的硬编码样式值。
- 涉及 action、event、focus handle、entity、context、global state 或 async task 时，先查对应技能目录，保持实现方式一致。
- 如果后续引入 `gpui-component` 或自定义设计系统，新增组件 API 要先追求命名稳定和可组合性，再追求一次性“做全”。
- 常规产品界面避免无意义卡片堆砌、过度装饰渐变和多重强调色；优先用版式、对齐、间距、字号、对比度和少量有目的的动效建立层级。

## 4. 提权与 GitHub 规则

- 任何需要联网、安装依赖、修改 `Cargo.lock`、执行 `cargo add` / `cargo install`、运行 `gh` 写操作，或其他超出沙盒权限的命令，都必须先申请权限。
- `gh` 在沙盒内通常不可用；无论是读操作还是写操作，只要因为沙盒、权限或认证环境导致失败，必须直接对原 `gh` 命令申请提权，不要先改用其它绕过方式。
- 如果命令因网络、权限或沙盒限制失败，应直接对原命令申请权限，不要改用绕过限制的替代方案。
- 删除文件、覆盖生成产物、批量改写文件或改写 lockfile 前，也要先说明并申请权限。
- GitHub 相关操作优先使用 `gh` 或已连接的 GitHub 工具，不要手写推测性结果。
- 编写 issue、PR、release note、评论前，如果仓库存在 `.github/` 下的模板、工作流或约定文件，先检查并遵循。
- 编写 PR 内容时，必须基于“当前分支相对远程最新 `main`”的整体差异进行总结，不要只根据最后一次提交或本次会话中的改动来写。
- 修改 README、LICENSE、仓库描述或其他公开信息时，要考虑 GitHub 仓库主页展示的一致性。

## 5. 验证与汇报

- 代码修改后，至少执行与改动直接相关的验证命令。
- 默认验证基线优先使用：
  - `cargo build`
  - `cargo test`
  - `cargo clippy --all-targets --all-features -- -D warnings`
- 如果改动只涉及文档、Agent 配置或仓库元数据，可以不运行构建，但汇报时要明确说明。
- 汇报结果时要写明实际执行过的命令；如果没执行，必须说明原因。

## 6. 仓库特例

- 当前主应用入口位于 `crates/kandb/src/main.rs`。
- 仓库还没有稳定模块分层；新增目录或 crate 前，先确认是否真的已经需要拆分，而不是为未来可能的复杂度提前设计。
- 由于项目还在早期阶段，产品方向相关文案允许先于功能实现存在，但必须明确标注为 roadmap 或 planned work，不能冒充既有能力。

## 7. 技能使用约定

- `.agents/skills/gpui-style-guide`：组件风格、命名、API 形状
- `.agents/skills/gpui-layout-and-style`：布局、样式、主题
- `.agents/skills/gpui-element`：低层 `Element` API
- `.agents/skills/gpui-entity`：实体状态与数据流
- `.agents/skills/gpui-async`：异步任务与后台工作
- `.agents/skills/gpui-action` / `gpui-event`：动作与事件处理
- `.agents/skills/gpui-focus-handle`：焦点与键盘交互
- `.agents/skills/gpui-context` / `gpui-global`：上下文和全局状态
- `.agents/skills/gpui-test`：GPUI 测试模式

当任务明显属于以上类别时，代理应先读取对应技能，再开始实现。
